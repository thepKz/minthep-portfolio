1. Các phương pháp phát triển phần mềm

1.1 Waterfall Model

Định nghĩa

Mô hình thác nước là phương pháp tuần tự, từng giai đoạn một, trong đó tiến độ phát triển được xem như chảy đều xuống qua các pha của dự án.

Quy trình thực hiện

Giai đoạn 1
Giai đoạn 2
Giai đoạn 3
Giai đoạn 4
Giai đoạn 5
Giai đoạn 6

Đặc điểm chính

  • Mỗi giai đoạn phải hoàn thành trước khi bắt đầu giai đoạn tiếp theo
  • Tài liệu chi tiết được tạo ra sau mỗi giai đoạn
  • Không có sự chồng chéo giữa các giai đoạn
  • Yêu cầu phải được xác định rõ ràng từ đầu

Ưu điểm

  • Dễ hiểu và quản lý
  • Tài liệu đầy đủ
  • Phù hợp với dự án nhỏ

Nhược điểm

  • Khó thay đổi yêu cầu
  • Rủi ro cao
  • Thời gian phát triển dài

3. Các loại kiểm thử

Kiểm thử chức năng

  • Unit Testing: Kiểm thử từng đơn vị code
  • Integration Testing: Kiểm thử tích hợp giữa các module
  • System Testing: Kiểm thử toàn bộ hệ thống
  • Acceptance Testing: Kiểm thử chấp nhận
  • Regression Testing: Kiểm thử hồi quy

Kiểm thử phi chức năng

  • Performance Testing: Kiểm thử hiệu năng
  • Security Testing: Kiểm thử bảo mật
  • Usability Testing: Kiểm thử sử dụng
  • Compatibility Testing: Kiểm thử tương thích
  • Load Testing: Kiểm thử tải
  • Stress Testing: Kiểm thử áp lực

Template cho Agile (Scrum)
Recommendation to use Agile:
I suggest using the Agile method for these reasons:

a) Requirements:
• Flexibility: Requirements not clearly defined, might change during development
• Adaptability: Need to adjust based on user feedback and market trends
• Thoroughness: Many requirements need early validation

b) Development Team:
• Team Structure: 4-6 experienced developers, perfect for Scrum
• Strong Collaboration: Good communication within team
• Role Expertise: Clear roles (PO, Scrum Master, Developers)
• Adaptability: Can quickly adjust to changes
• Time Constraints: Fits well with sprint cycles

Conclusion:
Scrum is best due to team structure, need for iteration, and ability to handle changing requirements.
Template cho Waterfall
Recommendation to use Waterfall:
I recommend the Waterfall method for these reasons:

a) Requirements:
• Clear and Stable: Requirements well-defined from start
• Structured Approach: Step-by-step process needed
• Detailed Planning: Complex requirements need thorough documentation

b) Development Team:
• Team Structure: Works well in structured environment
• Specific Roles: Clear separation of responsibilities
• Documentation Focus: Strong documentation practices
• Predictability: Well-defined timeline and milestones

Conclusion:
Waterfall is best due to clear requirements, structured roles, and need for documentation.

Câu 2: Testing Levels & Responsibilities (1-2đ)

Khung trả lời chuẩn

Based on the project requirements, I propose the following testing levels:

1. Unit Testing
   Tester: Development Team
   Focus: 
   - Individual components and functions
   - Basic functionality verification
   - Code coverage
   Tools: [JUnit, NUnit, etc.]

2. Integration Testing
   Tester: QA Team + Developers
   Focus:
   - Component interactions
   - Interface testing
   - Data flow between modules
   Approach: [Top-down/Bottom-up]

3. System Testing
   Tester: QA Team
   Focus:
   - End-to-end functionality
   - Performance testing
   - Security testing
   - Load testing
   Types: Both functional and non-functional

4. User Acceptance Testing (UAT)
   Tester: End Users/Stakeholders
   Focus:
   - Business requirements
   - User workflows
   - Real-world scenarios
   Approach: Black-box testing

Từ khóa quan trọng cho mỗi level

- White-box testing 		Developers During development

- Interface testing
- Data flow

- Performance

Tips quan trọng

    • Business tests → End users
    • Integration → Combined teams
  3. Giải thích lý do

    • How it helps project quality

Câu 4: User Stories (1.5đ)

Cấu trúc User Story chuẩn

Cấu trúc cơ bản:
As a [role]
I want to [action/feature]
So that [benefit/value]

Ví dụ:
As a train operator
I want to monitor the train's speed in real-time
So that I can ensure safe operation within speed limits
Các thành phần bắt buộc:
  • Role (Vai trò): Người dùng cụ thể
  • Action (Hành động): Chức năng cần thực hiện
  • Benefit (Lợi ích): Giá trị mang lại

INVEST Criteria cho User Stories

Independent
  • Có thể thực hiện độc lập
  • Không phụ thuộc stories khác
  • Tránh "và" trong mô tả
Negotiable
  • Có thể thảo luận
  • Không quá chi tiết
  • Để room cho đàm phán
Valuable
  • Mang lại giá trị rõ ràng
  • Có benefit cụ thể
  • Đáp ứng nhu cầu user
Estimable
  • Có thể ước lượng
  • Scope rõ ràng
  • Không quá lớn
Small
  • Đủ nhỏ cho một sprint
  • Có thể hoàn thành nhanh
  • Dễ quản lý
Testable
  • Có thể kiểm thử
  • Có tiêu chí verify
  • Đo lường được

Các Role thường gặp

Role Mô tả Ví dụ User Story
System Admin Quản trị hệ thống "As a system admin, I want to monitor system performance..."
End User Người dùng cuối "As an end user, I want to view my profile..."
Manager Quản lý "As a manager, I want to generate reports..."

Tips viết User Story hiệu quả

  1. Sử dụng ngôn ngữ đơn giản
    • Tránh thuật ngữ kỹ thuật
    • Viết ngắn gọn, rõ ràng
    • Dùng ngôn ngữ của user
  2. Tập trung vào giá trị
    • Mô tả rõ benefit
    • Liên kết với mục tiêu
    • Giải quyết nhu cầu thực
  3. Đảm bảo tính khả thi
    • Scope phù hợp
    • Có thể thực hiện được
    • Đo lường được kết quả

Loại Metric Mô tả Công thức/Cách tính Giá trị tốt

- RFC (Response For Class) 		- CBO: < 14

- NOC (Number of Classes) 		- LOC: Phụ thuộc dự án

Template trả lời mẫu

For measuring code and design quality, I propose the following metrics:

1. Cohesion Measurement
   Metric: LCOM (Lack of Cohesion of Methods)
   Description: Measures how well the methods of a class are related to each other
   Formula: [Insert formula]
   Threshold: Lower values indicate better cohesion
   Application: Used to identify classes that might need to be split

2. Coupling Analysis
   Metric: CBO (Coupling Between Objects)
   Description: Counts the number of other classes a class is dependent on
   Formula: [Insert formula]
   Threshold: < 14 is considered good
   Application: Helps identify overly dependent classes

3. Code Complexity
   Metric: Cyclomatic Complexity
   Description: Measures the number of linearly independent paths in code
   Formula: E - N + 2P (where E=edges, N=nodes, P=connected components)
   Threshold: < 10 for good maintainability
   Application: Identifies complex methods needing refactoring

4. Size Metrics
   Metric: Lines of Code (LOC)
   Description: Measures physical lines of code excluding comments
   Application: Helps estimate maintenance effort
   Threshold: Depends on function/class purpose

5. Inheritance Metrics
   Metric: Depth of Inheritance Tree (DIT)
   Description: Measures class hierarchy depth
   Threshold: < 6 levels
   Application: Prevents excessive inheritance complexity

Chi tiết từng loại Metric

Cohesion Metrics
  • LCOM (Lack of Cohesion of Methods)
    • Đo mức độ liên kết giữa các methods
    • Giá trị thấp = cohesion cao = tốt
    • Công thức: (M - sum(MV)/(M-1))
  • TCC (Tight Class Cohesion)
    • Đo tỷ lệ methods truy cập chung attributes
    • Giá trị > 0.5 là tốt
Coupling Metrics
  • CBO (Coupling Between Objects)
    • Đếm số lượng classes có liên kết
    • Giá trị < 14 là tốt
  • RFC (Response For Class)
    • Đếm số lượng methods có thể được gọi
    • Giá trị < 50 là tốt

Ví dụ Story Map cho "Train Schedule Management"

A. Activities and User tasks
1. Create Train Schedule
   1.1 Input Basic Schedule Information
   1.2 Set Route Details
   1.3 Assign Resources
   1.4 Validate Schedule

2. Update Train Schedule
   2.1 Modify Existing Schedule
   2.2 Handle Schedule Conflicts
   2.3 Notify Affected Parties
   2.4 Update Related Systems

1.1.1 As a scheduler, I want to input train departure times
1.1.2 As a scheduler, I want to input train arrival times
1.2.1 As a scheduler, I want to define train stops
2.1.1 As a scheduler, I want to modify schedule times
2.1.2 As a scheduler, I want to update route information

Release 2 (Advanced Features)
1.3.1 As a scheduler, I want to assign train staff
1.4.1 As a scheduler, I want to check schedule conflicts
2.2.1 As a scheduler, I want to resolve timing conflicts
2.3.1 As a scheduler, I want to send update notifications

Tips làm Story Map

Cấu trúc Activities
  • Chia hoạt động thành các nhóm logic
  • Sắp xếp theo thứ tự thực hiện
  • Đặt tên rõ ràng, ngắn gọn
  • Bao quát đầy đủ chức năng
Phân chia Releases
  • Ưu tiên chức năng cốt lõi cho Release 1
  • Phân bổ user stories hợp lý
  • Đảm bảo mỗi release có giá trị
  • Cân nhắc dependencies

Lỗi thường gặp

  • Sai cấu trúc đánh số
    • Activities phải bắt đầu từ 1, 2, 3...
    • Tasks phải có số thập phân (1.1, 1.2...)
    • User stories phải có 3 cấp số (1.1.1, 1.1.2...)
  • Thiếu logic trong phân chia
    • Activities không liên quan
    • Tasks không thuộc về activity
    • Stories không map với tasks
  • Sai format user stories
    • Thiếu cấu trúc As a/I want/So that
    • Stories không cụ thể
    • Không đáp ứng INVEST
